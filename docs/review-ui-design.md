# 审核界面设计

## 1. 目标

第一版审核界面只解决一个问题：让用户能查看 run 状态摘要，可视化查看图片、确认或修改标签、批量接受近重复组，并查看导出状态和结果路径。

不做：

- 登录和权限。
- 多人分配。
- 仲裁。
- 审核员绩效。
- 复杂质检工作流。
- 配置上传、配置编辑和启动 run。
- 通用标注平台能力。

## 2. 页面范围

### 任务列表

展示：

- run ID。
- 数据集名称。
- 标签体系。
- 总图片数。
- 自动采用数。
- 待审核数。
- 已修改数。
- 当前阶段。

操作：

- 进入审核。
- 查看报告。
- 查看导出状态和结果路径。

### 审核工作台

展示图片网格，支持筛选：

- 待审核。
- 低置信度。
- 质量风险。
- 近重复冲突。
- 离群候选。
- 自动采用抽样查看。
- 标签。
- 簇。

每个图片卡片展示：

- 缩略图。
- 推荐标签。
- 置信度。
- 质量状态。
- 审核状态。

### 单图详情

展示：

- 原图或大缩略图。
- image ID 和 URI。
- top-k 预标注候选。
- 质量原因。
- 相似图摘要。
- 所属簇代表样本。

操作：

- 接受推荐标签。
- 修改标签。
- 标记无效。
- 拒绝当前样本。
- 保存备注。

### 近重复组视图

展示：

- 组内图片。
- 代表图。
- 相似度。
- 推荐标签。
- 质量状态。

操作：

- 批量接受同一标签。
- 将冲突样本拆回单图审核。

### 简化簇视图

簇视图只用于辅助判断，不做复杂簇级工作流。

展示：

- 簇代表样本。
- 簇大小。
- 主标签占比。
- 预测分布。
- 离群候选。

操作：

- 查看簇内样本。
- 接受当前代表样本。
- 标记簇不纯。

第一版默认不允许整簇一键接受；如配置打开，也必须只接受高置信、非离群、质量为 `keep` 的子集。

## 3. 状态流转

初始状态：

- `pending`

自动决策可转为：

- `auto_accepted`

人工操作可转为：

- `human_accepted`
- `corrected`
- `rejected`
- `invalid`

自动采用样本被抽样查看后，如果用户修改标签，则从 `auto_accepted` 转为 `corrected`。

最终有效导出状态：

- `auto_accepted`
- `human_accepted`
- `corrected`

`rejected` 和 `invalid` 保留在完整结果中，但默认不作为有效标注样本。

## 4. 最小操作记录

第一版不追踪审核员责任，只记录结果所需的最小信息：

- `run_id`
- `image_id`
- `review_task_id`
- 原推荐标签。
- 最终标签。
- 操作类型。
- 操作时间。
- 备注。

这些信息用于界面回看、最终导出和简单错误统计。

## 5. 批量操作

近重复组批量接受条件：

- 同一 `duplicate_group_id`。
- 组内推荐标签一致。
- 组内图片质量状态为 `keep` 或配置允许的 `needs_review`。
- 组大小不超过 `review.batch_actions.max_batch_size`。
- 操作前展示影响数量。

批量操作落库规则：

- 后端将批量操作展开成多条单图 `review_decisions`。
- 每张图独立更新 `final_annotations`。
- 第一版不设计单独的 batch decision 表。

## 6. 抽样查看

自动采用结果可按配置抽样进入审核台，用于估计自动采用质量。

抽样维度：

- 标签。
- provider。
- 置信度分桶。
- 簇分布。
- 质量边界样本。

抽样查看不是复杂质检工作流。第一版只统计：

- 查看数量。
- 修改数量。
- 修改率。
- 主要错误标签对。

## 7. 报告

审核界面展示简要报告：

- 自动采用率。
- 待审核数量。
- 已修改数量。
- 每标签数量。
- 每标签修改率。
- 抽样查看修改率。
- 质量问题分布。
- 簇不纯标记数量。

审核界面只提供判断依据，不自动定义任务是否完成。是否可以导出由用户根据这些摘要人工决定；导出动作仍通过 CLI 触发。
